# Architectural Decisions & Rationale

## 1. Bluetooth (BLE) Architecture
- **Shared Controller**: We use a modular BLE architecture where `BleConnection` handles the low-level GATT logic and `TreadmillManager` / `HeartRateManager` handle the high-level protocol parsing.
- **Mocks**: Every manager has a mock implementation (`MockTreadmillManager`) to allow UI and logic development without physical hardware.
- **Pairing List Semantics**: BLE scan results are deduplicated by device address. A device is hidden until it has multiple scan events, then qualified devices are revealed gradually in a stable order. Reveal priority uses scan event count first and average RSSI second; once a device appears, later RSSI updates do not move its row. Pairing and profile surfaces show selected and connected status separately because a saved device is not necessarily connected.
- **Explicit Device Role Selection**: Selecting a BLE device opens a neutral role-choice bottom sheet for WalkingPad or Watch. The app saves the chosen role explicitly instead of inferring it from the advertised name, and if the same address was previously saved under the other role it is cleared from that slot.
- **Saved Device Reconnect**: Session and profile surfaces attempt to reconnect saved treadmill/watch addresses when opened so the app can recognize devices that are already paired with Android.
- **Reconnect Cleanup**: Managers close any existing GATT connection before connecting to a new saved or selected address.
- **Profile WalkingPad Controls**: The Profile WalkingPad row includes a `BT` action that starts Bluetooth connection for the saved pad when disconnected and a refresh action that reconnects the saved pad to rerun service discovery and the KS protocol handshake.
- **BLE Operation Serialization**: Android BLE allows only one reliable in-flight GATT operation at a time. `BleConnection` owns a single operation mutex for descriptor writes and characteristic writes, starts descriptor/characteristic writes on the main thread with the legacy `setValue`/`writeCharacteristic` and descriptor APIs used by KS Fit's BLE stack, waits for write callbacks for acknowledged operations, applies a small settle delay for write-without-response operations, and completes pending operations on disconnect. Higher-level managers can still coalesce user intent, but the controller is the source of truth for avoiding GATT operation races.
- **WalkingPad BLE Control Loop**: The WalkingPad does not rely on unsolicited status updates. The pad connection forces BLE transport, refreshes Android's GATT cache, delays initial service discovery, retries when Android never delivers the service-discovery callback, and then inspects the full discovered GATT table for either the legacy `FE00` WalkingPad service or an encrypted KS read/write characteristic pair. If neither KS protocol path can be found, the app reports the device as unsupported instead of treating generic treadmill services as usable. For legacy pads, the manager enables notifications on `FE01`, writes commands to `FE02` without response when Android exposes that property, waits briefly for the protocol to settle, polls current status about every 750 ms, and serializes outbound commands with short spacing so preference sync, start/stop, mode, and speed commands are not sent back-to-back. For newer encrypted KS pads, the manager discovers the APK-matched read/write characteristic pairs (`FED8/FED7`, `0001FED7/0001FED8`, `0002FED8/0002FED7`, `24e252...e/24e252...f`, or AIS `5833ff03/5833ff02`), verifies that the read candidate supports notify/indicate and the write candidate supports write/write-no-response, completes the KS text handshake, then polls `servers getProp` for the KS Fit APK-confirmed motion/state properties: `RunningSteps`, `BurnCalories`, `CurrentSpeed`, `RunningDistance`, `RunningTotalTime`, `ControlMode`, and `runState`. The `5833ff01` service uses Alibaba AIS framing, so C3P0 wraps outbound encrypted KS text in AIS general-command frames and unwraps/reassembles AIS responses before decoding KS text; because KS Fit's BLE sub-version 18 path prefixes AIS command payloads with one bus byte, the `5833ff02` handshake starts with that bus prefix and toggles to unprefixed AIS only if the prefixed handshake does not answer. Delayed encrypted handshake and AIS fallback writes are stamped with the current connection generation and skipped if Android disconnects the GATT, the unknown-table handshake candidate burst is paced to avoid Android GATT `133` write failures on write-with-response characteristics, and the BLE controller refuses characteristic writes unless the underlying GATT is still connected. C3P0 intentionally supports only KS WalkingPad protocols so commands, counters, pause/stop behavior, and mode telemetry can target this specific pad instead of falling back to generic treadmill control. The pad is only reported as connected once at least one supported KS protocol path is ready; speed and visible mode controls stay disabled until then, and the protocol readiness watchdog starts only after service discovery finds a supported KS path so `services=none` points to a GATT discovery failure instead of a KS handshake failure. The legacy command bytes, checksum, and status packet layout are aligned with the ph4-walkingpad, QWalkingPad, and Go WalkingPad controllers; belt state is decoded separately from manual/auto mode (`0` stopped, `1` active, `5` standby, `9` starting). Native WalkingPad automatic mode remains a parsed device mode but is not exposed in the session UI.
- **KingSmith FTMS Supplement Path**: When a pad exposes the KingSmith FTMS service (`0x1826`), Control Point (`0x2ad9`), Treadmill Data (`0x2acd`), and the KS ODM/supplement write channel (`d18d2c10` or `24e2521c...d/f00fdf7`), C3P0 treats that as a KS-specific protocol path and prefers it over the generic-looking `5833ff01` service. The app enables staggered FTMS notifications, writes KS Fit's property-list preamble (`01 00 0d 00 06 0b 0f 0d`) through the ODM/supplement channel, sends basic start/pause/speed through FTMS Control Point, sends final stop as FTMS `08 01`, sends pause as FTMS `08 02`, and parses KingSmith's bit-13 Treadmill Data step extension so WalkingPad steps come from the pad.
- **Bluetooth Error Visibility**: BLE scan, connection, service discovery, notification, command-write failures, missing WalkingPad status replies, and WalkingPad commands that do not appear in subsequent status are reported through a singleton error reporter and shown as persistent stacked error cards in the root UI. Errors do not auto-dismiss, making transient asynchronous failures visible for troubleshooting.
- **WalkingPad Command Bounds**: Speed changes are clamped in the UI, auto-speed controller handoff, mock manager, and real BLE manager so no caller can send an out-of-range speed.
- **WalkingPad Moving Speed Floor**: Normal manual and Zone 2 speed requests floor to 1 mph (`1.60934 km/h`) because the walking belt can stop when commanded below that threshold. Explicit stop remains a separate zero-speed/stop command path.
- **WalkingPad Speed Step Units**: Manual +/- speed controls step by the displayed unit: `0.1 mph` in imperial mode and `0.1 km/h` in metric mode, with commands still sent internally in km/h.
- **WalkingPad Steps And Calories**: Legacy KS status packets expose a step counter directly, and encrypted KS props expose `RunningSteps` plus `BurnCalories`. Calories are taken from the WalkingPad counter delta; Health Connect weight is read only for the Stats trend chart and is not used to estimate calories.
- **WalkingPad Start Timing**: Session start sends the pad's start/resume command before applying the safe moving-speed target. The speed target is delayed until after the pad's hardware countdown so the belt does not begin moving before the countdown completes.
- **WalkingPad Command Responsiveness**: User actions avoid avoidable preflight writes. Session controls do not force a unit sync before every action, speed commands use a shorter BLE spacing than structural commands, and rapid +/- taps advance from the app's pending target so they coalesce to the newest requested speed instead of waiting for treadmill status packets or queueing stale intermediate speeds.
- **WalkingPad Native Automatic Mode**: Native Automatic is still understood through legacy KS `FE00` mode telemetry and the reverse-engineered encrypted KS `ControlMode` values, but it is hidden from the user-facing session and default-mode controls. Encrypted mode switching still uses `props ControlMode 1/2` for Manual/Standby after the KS encrypted handshake identifies the active encryption table.
- **Zone 2 HR Safety**: Zone 2 mode is only available with fresh heart-rate data. If HR becomes stale while Zone 2 is active, the app exits Zone 2, switches the pad to Manual, and sets the belt to the 1 mph moving floor.
- **Zone 2 Control Tuning**: Zone 2 speed control decides every 15 seconds and uses stepped speed changes while outside the target HR band: roughly 0.1, 0.2, 0.3, 0.4, or 0.5 mph depending on how far HR is outside Zone 2. The 0.5 mph jump is reserved for large misses, and learned in-zone speeds cannot force a larger jump than the current HR-error band allows. Once inside Zone 2 it does not chase the center. It only applies a slow edge guard when HR is within the outer 5% of the zone band, with a minimum 2 bpm guard band, and the recent 2-minute trend points out of Zone 2. Edge nudges are 0.1 mph and occur no more than once every 45 seconds.
- **Zone 2 Max Speed**: Profile stores a user-configurable Zone 2 maximum speed, defaulting to 3.5 mph. The setting is saved internally in km/h, displayed in the selected unit system, passed into the Zone 2 controller, and also applied at the final treadmill speed command clamp.
- **Heart Rate Chart Scaling**: The live heart-rate chart scales to the observed session HR range plus the user's Zone 2 min/max guide lines, with 5% padding around the chosen bounds. With little or no HR history, Zone 2 provides the fallback range so the guide lines stay visible and the chart does not waste vertical space.
- **Default Session Mode**: Profile settings choose Manual or Zone 2 for cold session starts. The default stored value is Zone 2. Zone 2 falls back to Manual if heart-rate data is not fresh. A legacy stored Automatic value no longer maps to an enum value and falls back to Zone 2.
- **Active Session Wake Lock**: The optional keep-awake setting applies only while a session is active and unpaused. Paused sessions intentionally allow the phone to sleep.
- **KS Stop Sequence**: The KS app treats pause and stop as separate states: pause is allowed while the belt is moving, but final stop is only valid once the belt has coasted to zero. C3P0 follows the KS-specific path by sending FTMS pause as `08 02`, FTMS final stop as `08 01`, legacy zero-speed commands on `FE02`, or encrypted `props runState 0` commands over `FED7`/`FED8`-style characteristics in 16-byte chunks. Some pads can acknowledge a stopped UI state while retaining a nonzero belt target, so stop logic is verified against subsequent KS status instead of assuming a write succeeded.
- **WalkingPad Counter Decoding**: Status notifications decode 24-bit time, distance, and step counters explicitly to avoid operator-precedence ambiguity.

## 2. Session Management
- **Foreground Service**: Sessions run in `SessionService` to prevent the OS from killing the app during long workouts. This is required for reliable BLE data collection and Health Connect exports.
- **Persistence**: Real-time metrics (HR, Speed, Cadence) are saved to Room as they arrive, ensuring data isn't lost if the app crashes.
- **Pause Semantics**: Starting a session does not mark the app active or create the Room session until the WalkingPad start command succeeds, so failed pad starts cannot leave an open app-local workout. Pausing a session stops the WalkingPad, pauses active stat collection, skips Room metric writes, and excludes paused HR readings from the average. Resuming restarts the WalkingPad only if the start command succeeds and continues the same session without resetting stats. The live elapsed session stat uses the WalkingPad's reported running-time counter delta instead of an app-local timer.
- **Stop Safety**: Stopping a session requires a 3-second circular long press with a progress ring to prevent accidental session loss. While held, the stop button scales up in place with high elevation so the ring remains visible around the user's finger.
- **Device Commands**: Session start sends WalkingPad start, pause sends stop, resume sends start, and stop captures final counters before sending stop.
- **WalkingPad Modes**: Session controls show Manual and Zone 2. Manual maps directly to the WalkingPad's native manual mode. Zone 2 is app-controlled: the app keeps the WalkingPad in manual mode and adjusts speed itself from heart-rate feedback. Native WalkingPad automatic mode is parsed for status compatibility but is intentionally not shown as a selectable session mode. The Zone 2 controller keeps a five-minute memory of recent speed/heart-rate outcomes, including speeds that produced Zone 2, so low-HR corrections can move above a recently learned Zone 2 belt speed instead of only applying a blind proportional nudge.
- **Session Aggregates**: Stored session totals use deltas from the start counters. Calories use the WalkingPad calorie counter delta. HR aggregates come from active readings only. Cadence uses step deltas over elapsed metric time.
- **Live Session Stats**: The session dashboard also displays distance and steps as deltas from the session start counters, matching persisted session totals.
- **Step Goal ETA**: The session dashboard estimates time remaining to the daily step goal from the current session's average steps per minute and the Health Connect steps remaining. The ETA stays blank until the session has at least 30 seconds and 10 steps of data, and shows `Done` once steps remaining reaches zero.
- **Inactive Devices**: Starting/resuming with an inactive watch, stale/missing HR broadcast, or inactive WalkingPad shows a bottom sheet with Pair, Continue, and Never Ask Again options. A connected watch is not considered HR-ready until a positive heart-rate sample has arrived recently.
- **Time Display Caveat**: Current session time is pad-reported active/moving time. If a separate wall-clock elapsed time is needed, add a second stat instead of changing active time semantics.

## 3. Data Strategy
- **Health Connect**: Chosen over Google Fit as the primary health data hub for modern Android devices. We write completed walking sessions, steps, and distance, read aggregate steps for history, and read weight history for the Stats trend chart. We do not request heart-rate records; the WalkingPad owns calories and the watch remains the source of HR data in Health Connect.
- **Health Connect Permissions**: The profile toggle reflects actual granted permissions. Enabling shows the Health Connect permission flow; disabling opens Health Connect settings because apps cannot revoke Health Connect grants directly. Read/write code paths check only the permissions needed for that operation. Step history requests both step-read access and Health Connect historical-data access so periods older than the default recent-read window, such as May data on a July install, can be returned when the user grants it.
- **Historical Steps**: Daily Health Connect history uses Health Connect aggregate steps directly. Health Connect's app priority and aggregation behavior deduplicates overlapping step sources, so C3P0 does not read record-level step origins or perform app-side overlap subtraction.
- **DataStore**: Used for lightweight persistence like BLE device addresses, unit preferences, inactive-warning preference, backup toggle, profile age, daily step goal, default session mode, and WalkingPad preferences. Room handles the heavy lifting for session history.
- **No-Load Stop Preference**: The Profile screen exposes KS Fit's no-load stop options: off/on plus 5, 15, 30, 45, or 60 seconds. KS Fit 6.0.2's Flutter binary exposes `AuToStop` as the switch key and `NoloadStop` as the timeout key, so encrypted KS text transports send `props AuToStop 0/1` and `props NoloadStop <seconds>`. FTMS+supplement pads subscribe to known supplement notify candidates and send the supplement `autoStop` property id (`0x02`) write with boolean value `0/1`; hardware testing captured `72 81 02 02 00 f7` as the disabled response and showed that sending timeout seconds as the `autoStop` value is rejected. C3P0 treats the setting as saved only when a new `autoStop` response matches the requested on/off value. The numeric `servers getProp` IDs for proactive polling and the FTMS timeout setter were not recovered; C3P0 parses named props if the pad echoes them and reports unmatched FTMS bytes plus any supplement notification for sniff-based follow-up.
- **Unit Source of Truth**: The app's stored unit setting is the source of truth. WalkingPad units are corrected when settings change, when a session starts/resumes/pauses, and when reported device units diverge.
- **KS Unit Handling**: The app's unit setting is synced through the KS protocol where supported. Legacy WalkingPad unit preference commands use the pad's proprietary command path, and encrypted KS telemetry reports the current device unit preference for drift correction.
- **Auto Backup**: We rely on Android's native Google Drive Auto Backup to sync Room DB files and DataStore preferences. Backup rules include `c3p0.db`, `c3p0.db-shm`, `c3p0.db-wal`, `datastore/settings.preferences_pb`, and SharedPreferences.
- **Backup Requests**: When Google Drive backup is enabled, the app calls `BackupManager.dataChanged()` on cold start, persisted settings changes, active session progress on a 5-minute throttle, and session completion. This requests Android to schedule a backup; it does not force an immediate upload or restore.
- **Backup Throttle**: Throttled backup requests check the in-memory interval before reading DataStore, and the first request after process start is allowed even when a throttle is supplied.

## 4. UI/UX
- **Jetpack Compose**: Used for all UI. We opted for a simple Dashboard with "Status Dots" to provide immediate feedback on device connectivity.
- **Auto-Speed Algorithm**: Implemented as a Proportional Controller with a Moving Average to ensure treadmill adjustments are smooth and not jittery.
- **Permission Rationale**: Code paths that require Android permissions should show a bottom sheet explaining the request before invoking the system prompt.
- **Session Dashboard Density**: Session stats use a compact adaptive grid: up to three tiles per row when the available phone width can keep each tile readable, falling back to two columns on narrower screens. The metrics, HR chart, and speed controls live in a scrollable session body, while the start/pause/stop controls and Manual/Zone 2 selector stay docked in a bottom footer so the primary controls remain visible. Steps remaining uses today's Health Connect baseline plus in-progress session steps.
- **Speed Readout Color**: The session speed readout uses the normal surface-variant text color instead of bright green so it stays readable without visual glare.
- **Heart-Rate Chart**: The session chart includes y-axis labels and zone-colored line segments for Zone 0 through Zone 4.
- **Shared Chart Controls**: Stats charts use a reusable period-and-grouping control that renders horizontal range chips and a grouping dropdown from chart-specific option lists. Chart types keep their own date-window and grouping semantics while sharing the UI behavior and styling.
- **Stats History**: The History screen shows compact tile-sized preview cards for Health Connect steps and weight trend, arranged two per row, and opens the full chart controls in a wrap-content bottom sheet when a preview is tapped. The steps preview and full steps chart show single Health Connect aggregate step bars; the full chart has 1W/1M/3M/6M/1Y range chips plus a Day/Week/Month grouping menu. Step goal guides and green goal-met bars use the matching grouping threshold: daily goal for days, daily goal times seven for weeks, and daily goal times the calendar month's day count for months. The full steps chart draws the goal as a full-width dashed line with a fixed y-axis label, sizes bar slots from the selected period so shorter ranges show their whole visible window without capping scrollable history, and scales y-height from the current viewport plus one selected period before and after. The chart trims only leading empty periods, keeps zero-step periods after activity begins, right-aligns short timelines, opens on the latest periods, and scrolls left through the retained one-year step cache. Step and weight history are cached in `noBackupFilesDir` so the last successful history appears immediately while a Health Connect refresh runs, with the Refresh action replaced by a spinner during the read. The screen also requests Health Connect weight history from the weight card and charts daily average raw weight against a 7-day trailing average; the weight preview shows the latest 7-day average value while charting both raw weight and the 7-day trend with smoothed rounded curves, and the full weight chart uses smoothed raw/trend curves, a single quick-options row for 1W/1M/3M/6M/1Y plus D/W/M grouping, supports custom pinch-adjusted x-axis windows, labels the x-axis from the current viewport, dynamically scales the y-axis and summary average/difference to the currently visible x-axis window, and long-press/drag inspection shows exact grouped raw and trend values with matching line/tooltip colors and per-point haptic feedback.

## 5. CI & Distribution
- **Debug APK Workflow**: GitHub Actions builds a debug APK on every push and uploads it as an artifact for remote install testing. Debug builds use the checked-in `app/debug.keystore` so GitHub artifacts and local debug installs share the same package signature and can install over each other. Release APKs are deferred until signing keys exist.
- **Artifact Retention**: CI artifacts should use a finite retention period because artifact storage can count against repository/account limits.
- **Launcher Icon Assets**: C3P0 uses Android adaptive launcher icons on modern devices through the `@mipmap/c3p0_launcher` manifest icon resource. The adaptive icon background is a solid dark blue vector, and the foreground is a generated XML vector version of the robot art with its smaller size baked into the path coordinates so Pixel Launcher can mask the icon without clipping the robot or placing the supplied square PNG inside a white legacy icon circle. The manifest intentionally avoids a separate `roundIcon` path so launchers resolve one icon source, while supplied density PNGs remain checked in as legacy fallbacks.
