# Architectural Decisions & Rationale

## 1. Bluetooth (BLE) Architecture
- **Shared Controller**: We use a modular BLE architecture where `BleConnection` handles the low-level GATT logic and `TreadmillManager` / `HeartRateManager` handle the high-level protocol parsing.
- **Mocks**: Every manager has a mock implementation (`MockTreadmillManager`) to allow UI and logic development without physical hardware.
- **Pairing List Semantics**: BLE scan results are deduplicated by device address. A device is hidden until it has multiple scan events, then qualified devices are revealed gradually in a stable order. Reveal priority uses scan event count first and average RSSI second; once a device appears, later RSSI updates do not move its row. Pairing and profile surfaces show selected and connected status separately because a saved device is not necessarily connected.
- **Explicit Device Role Selection**: Selecting a BLE device opens a neutral role-choice bottom sheet for WalkingPad or Watch. The app saves the chosen role explicitly instead of inferring it from the advertised name, and if the same address was previously saved under the other role it is cleared from that slot.
- **Saved Device Reconnect**: Session and profile surfaces attempt to reconnect saved treadmill/watch addresses when opened so the app can recognize devices that are already paired with Android.
- **Reconnect Cleanup**: Managers close any existing GATT connection before connecting to a new saved or selected address.
- **WalkingPad BLE Control Loop**: The WalkingPad does not rely on unsolicited status updates. The pad connection forces BLE transport, refreshes Android's GATT cache, delays initial service discovery, and then inspects the full discovered GATT table for either the legacy `FE00` WalkingPad service or an encrypted KS read/write characteristic pair. If neither KS protocol path can be found, the app reports the device as unsupported instead of treating generic treadmill services as usable. For legacy pads, the manager enables notifications on `FE01`, writes commands to `FE02` without response when Android exposes that property, waits briefly for the protocol to settle, polls current status about every 750 ms, and serializes outbound commands with short spacing so preference sync, start/stop, mode, and speed commands are not sent back-to-back. For newer encrypted KS pads, the manager discovers the APK-matched read/write characteristic pairs (`FED8/FED7`, `0001FED7/0001FED8`, or `0002FED8/0002FED7`), enables notification or indication updates based on characteristic properties, completes the KS text handshake, then polls `servers getProp` for the KS Fit APK-confirmed motion/state properties: `RunningSteps`, `BurnCalories`, `CurrentSpeed`, `RunningDistance`, `RunningTotalTime`, `ControlMode`, and `runState`. C3P0 intentionally supports only KS WalkingPad protocols so commands, counters, pause/stop behavior, and Automatic mode can target this specific pad instead of falling back to generic treadmill control. The pad is only reported as connected once at least one supported KS protocol path is ready; speed and mode controls stay disabled until then, and readiness timeout errors include Android's discovered BLE services. The legacy command bytes, checksum, and status packet layout are aligned with the ph4-walkingpad, QWalkingPad, and Go WalkingPad controllers; belt state is decoded separately from manual/auto mode (`0` stopped, `1` active, `5` standby, `9` starting). Native WalkingPad automatic mode is exposed only when the connected KS protocol supports the mode command.
- **Bluetooth Error Visibility**: BLE scan, connection, service discovery, notification, command-write failures, missing WalkingPad status replies, and WalkingPad commands that do not appear in subsequent status are reported through a singleton error reporter and shown as persistent stacked error cards in the root UI. Errors do not auto-dismiss, making transient asynchronous failures visible for troubleshooting.
- **WalkingPad Command Bounds**: Speed changes are clamped in the UI, auto-speed controller handoff, mock manager, and real BLE manager so no caller can send an out-of-range speed.
- **WalkingPad Moving Speed Floor**: Normal manual and Zone 2 speed requests floor to 1 mph (`1.60934 km/h`) because the walking belt can stop when commanded below that threshold. Explicit stop remains a separate zero-speed/stop command path.
- **WalkingPad Speed Step Units**: Manual +/- speed controls step by the displayed unit: `0.1 mph` in imperial mode and `0.1 km/h` in metric mode, with commands still sent internally in km/h.
- **WalkingPad Steps And Calories**: Legacy KS status packets expose a step counter directly, and encrypted KS props expose `RunningSteps` plus `BurnCalories`. If the pad does not expose calories through the active KS path, calories fall back to the existing body-weight/speed estimate.
- **WalkingPad Start Timing**: Session start sends the pad's start/resume command before applying the safe moving-speed target. The speed target is delayed until after the pad's hardware countdown so the belt does not begin moving before the countdown completes.
- **WalkingPad Command Responsiveness**: User actions avoid avoidable preflight writes. Session controls do not force a unit sync before every action, speed commands use a shorter BLE spacing than structural commands, and rapid +/- taps advance from the app's pending target so they coalesce to the newest requested speed instead of waiting for treadmill status packets or queueing stale intermediate speeds.
- **WalkingPad Native Automatic Mode**: Native Automatic is supported through legacy KS `FE00` mode commands when available and through the reverse-engineered encrypted KS command path when `FED7`/`FED8`-style characteristics are present. Encrypted mode switching sends `props ControlMode 0/1/2` for Automatic/Manual/Standby after the KS encrypted handshake identifies the active encryption table.
- **Zone 2 HR Safety**: Zone 2 mode is only available with fresh heart-rate data. If HR becomes stale while Zone 2 is active, the app exits Zone 2, switches the pad to Manual, and sets the belt to the 1 mph moving floor.
- **Zone 2 Control Tuning**: Zone 2 speed control decides every 15 seconds and uses stepped speed changes while outside the target HR band: roughly 0.1, 0.2, 0.3, 0.4, or 0.5 mph depending on how far HR is outside Zone 2. The 0.5 mph jump is reserved for large misses, and learned in-zone speeds cannot force a larger jump than the current HR-error band allows. Once inside Zone 2 it does not chase the center. It only applies a slow edge guard when HR is within the outer 5% of the zone band, with a minimum 2 bpm guard band, and the recent 2-minute trend points out of Zone 2. Edge nudges are 0.1 mph and occur no more than once every 45 seconds.
- **Zone 2 Max Speed**: Profile stores a user-configurable Zone 2 maximum speed, defaulting to 3.5 mph. The setting is saved internally in km/h, displayed in the selected unit system, passed into the Zone 2 controller, and also applied at the final treadmill speed command clamp.
- **Heart Rate Chart Scaling**: The live heart-rate chart scales to the observed session HR range plus the user's Zone 2 min/max guide lines, with 5% padding around the chosen bounds. With little or no HR history, Zone 2 provides the fallback range so the guide lines stay visible and the chart does not waste vertical space.
- **Default Session Mode**: Profile settings choose the default mode for cold session starts. The default stored value is Zone 2. Zone 2 falls back to Manual if heart-rate data is not fresh, and Automatic falls back to Manual if native WalkingPad automatic mode is unavailable.
- **Active Session Wake Lock**: The optional keep-awake setting applies only while a session is active and unpaused. Paused sessions intentionally allow the phone to sleep.
- **KS Stop Sequence**: The KS app treats pause and stop as separate states: pause is allowed while the belt is moving, but final stop is only valid once the belt has coasted to zero. C3P0 follows the KS-specific path by sending legacy zero-speed commands on `FE02` or encrypted `props runState 0` commands over `FED7`/`FED8`-style characteristics in 16-byte chunks. Some pads can acknowledge a stopped UI state while retaining a nonzero belt target, so stop logic is verified against subsequent KS status instead of assuming a write succeeded.
- **WalkingPad Counter Decoding**: Status notifications decode 24-bit time, distance, and step counters explicitly to avoid operator-precedence ambiguity.

## 2. Session Management
- **Foreground Service**: Sessions run in `SessionService` to prevent the OS from killing the app during long workouts. This is required for reliable BLE data collection and Health Connect exports.
- **Persistence**: Real-time metrics (HR, Speed, Cadence) are saved to Room as they arrive, ensuring data isn't lost if the app crashes.
- **Pause Semantics**: Pausing a session stops the WalkingPad, pauses active stat collection, skips Room metric writes, and excludes paused HR readings from the average. Resuming restarts the WalkingPad and continues the same session without resetting stats. The live elapsed session stat uses the WalkingPad's reported running-time counter delta instead of an app-local timer.
- **Stop Safety**: Stopping a session requires a 3-second circular long press with a progress ring to prevent accidental session loss. While held, the stop button scales up in place with high elevation so the ring remains visible around the user's finger.
- **Device Commands**: Session start sends WalkingPad start, pause sends stop, resume sends start, and stop captures final counters before sending stop.
- **WalkingPad Modes**: Session controls always show Manual, Automatic, and Zone 2 so unavailable capabilities are visible rather than disappearing. Automatic is enabled only when the connection has the native KS mode command available, and remains standby-only in the UI and manager, matching KS Fit's behavior that mode switching is unavailable while the treadmill is actively running. Manual and Automatic map directly to the WalkingPad's native manual/automatic modes. Zone 2 is app-controlled: the app keeps the WalkingPad in manual mode and adjusts speed itself from heart-rate feedback. The Zone 2 controller keeps a five-minute memory of recent speed/heart-rate outcomes, including speeds that produced Zone 2, so low-HR corrections can move above a recently learned Zone 2 belt speed instead of only applying a blind proportional nudge.
- **Session Aggregates**: Stored session totals use deltas from the start counters. Calories use a MET estimate based on active duration, body weight, and average speed. HR aggregates come from active readings only. Cadence uses step deltas over elapsed metric time.
- **Live Session Stats**: The session dashboard also displays distance and steps as deltas from the session start counters, matching persisted session totals.
- **Inactive Devices**: Starting/resuming with an inactive watch, stale/missing HR broadcast, or inactive WalkingPad shows a bottom sheet with Pair, Continue, and Never Ask Again options. A connected watch is not considered HR-ready until a positive heart-rate sample has arrived recently.
- **Time Display Caveat**: Current session time is pad-reported active/moving time. If a separate wall-clock elapsed time is needed, add a second stat instead of changing active time semantics.

## 3. Data Strategy
- **Health Connect**: Chosen over Google Fit as the primary health data hub for modern Android devices. We write completed walking sessions, steps, and distance. We read body weight for calorie estimates and steps for normalization. We do not request or write heart-rate records; the watch should remain the source of HR data in Health Connect.
- **Health Connect Permissions**: The profile toggle reflects actual granted permissions. Enabling shows the Health Connect permission flow; disabling opens Health Connect settings because apps cannot revoke Health Connect grants directly. Read/write code paths check only the permissions needed for that operation.
- **Normalized Historical Steps**: Daily Health Connect history uses Health Connect aggregate steps for raw daily totals, then subtracts non-C3P0 step records that overlap completed C3P0 sessions. Record-level counts are prorated across day/session boundaries for the C3P0 and excluded-source breakdown.
- **DataStore**: Used for lightweight persistence like BLE device addresses, unit preferences, inactive-warning preference, backup toggle, profile age, daily step goal, and cached body weight. Room handles the heavy lifting for session history.
- **No-Load Stop Preference**: No-load stop controls are intentionally hidden until the proprietary KS Fit BLE command/key is captured. The public WalkingPad protocol references reviewed so far do not publish the proprietary key, so the app does not present a setting it cannot apply to the pad.
- **Unit Source of Truth**: The app's stored unit setting is the source of truth. WalkingPad units are corrected when settings change, when a session starts/resumes/pauses, and when reported device units diverge.
- **KS Unit Handling**: The app's unit setting is synced through the KS protocol where supported. Legacy WalkingPad unit preference commands use the pad's proprietary command path, and encrypted KS telemetry reports the current device unit preference for drift correction.
- **Auto Backup**: We rely on Android's native Google Drive Auto Backup to sync Room DB files and DataStore preferences. Backup rules include `c3p0.db`, `c3p0.db-shm`, `c3p0.db-wal`, `datastore/settings.preferences_pb`, and SharedPreferences.
- **Backup Requests**: When Google Drive backup is enabled, the app calls `BackupManager.dataChanged()` on cold start, persisted settings changes, active session progress on a 5-minute throttle, and session completion. This requests Android to schedule a backup; it does not force an immediate upload or restore.
- **Backup Throttle**: Throttled backup requests check the in-memory interval before reading DataStore, and the first request after process start is allowed even when a throttle is supplied.

## 4. UI/UX
- **Jetpack Compose**: Used for all UI. We opted for a simple Dashboard with "Status Dots" to provide immediate feedback on device connectivity.
- **Auto-Speed Algorithm**: Implemented as a Proportional Controller with a Moving Average to ensure treadmill adjustments are smooth and not jittery.
- **Permission Rationale**: Code paths that require Android permissions should show a bottom sheet explaining the request before invoking the system prompt.
- **Session Dashboard Density**: Session stats use a compact adaptive grid: up to three tiles per row when the available phone width can keep each tile readable, falling back to two columns on narrower screens. This keeps distance, steps, normalized steps remaining to goal, active time, calories, current HR, average HR, speed controls, device status, and the HR chart visible without forcing vertical scrolling on typical session screens. Steps remaining uses today's normalized Health Connect baseline plus in-progress session steps. Session action controls are centered to avoid the control row feeling lopsided.
- **Speed Readout Color**: The session speed readout uses the normal surface-variant text color instead of bright green so it stays readable without visual glare.
- **Heart-Rate Chart**: The session chart includes y-axis labels and zone-colored line segments for Zone 0 through Zone 4.
- **Stats History**: The History screen shows Health Connect raw/normalized step history as a compact horizontal bar chart with day, week, and month grouping, plus selected-session details with stored aggregates, metric-derived fallback summaries, and Health Connect normalized step counts when available. The chart is constrained to roughly a quarter of the page, opens on the latest periods, and scrolls left to review older periods.

## 5. CI & Distribution
- **Debug APK Workflow**: GitHub Actions builds a debug APK on every push and uploads it as an artifact for remote install testing. Release APKs are deferred until signing keys exist.
- **Artifact Retention**: CI artifacts should use a finite retention period because artifact storage can count against repository/account limits.
